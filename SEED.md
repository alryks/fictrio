Нужно реализовать реалистичное синтетическое наполнение базы данных пользовательским поведением.

Контекст:
- В каталоге уже загружаются фильмы, сериалы, сезоны, эпизоды и книги.
- Фильмов, сериалов и книг примерно по 1000.
- Сезонов и эпизодов уже больше 1000.
- В таблице roles пока оставляем только 3 роли. Не нужно искусственно создавать 1000 ролей.
- В данных по фильмам, сериалам и книгам есть средняя оценка и количество оценок из внешних источников. Их нужно использовать при генерации поведения пользователей. Находятся они в src/data/
- Цель — не просто набить таблицы строками, а создать правдоподобную социальную активность: оценки, отзывы, комментарии, списки, элементы списков, подписки, прогресс и действия модерации.

Задача:
Реализуй генератор seed-данных, который создаёт реалистичное поведение пользователей и заполняет социальные таблицы БД. Генератор должен быть детерминированным при фиксированном seed, идемпотентным или безопасно перезапускаемым, быстрым и не нарушающим ограничения внешних ключей, unique-ограничения и бизнес-ограничения БД.

Нужно заполнить минимум:
- users: 1000+ пользователей;
- user_roles: роли для пользователей;
- ratings: желательно 30 000–100 000 записей;
- posts: желательно 5 000–20 000 записей, включая отзывы и комментарии;
- lists: желательно 2 000–8 000 списков;
- list_items: желательно 20 000–80 000 элементов списков;
- follows: желательно 5 000–30 000 подписок;
- progress: желательно 10 000–50 000 записей;
- moderation_actions: минимум 1000 записей;
- roles: оставить 3 записи, не раздувать до 1000.

Важно:
Не генерируй данные равномерно. Поведение должно быть похоже на реальное.

Реализуй модель пользовательских профилей активности. Например:
1. inactive — неактивные пользователи, почти без действий.
2. observer — наблюдатели, иногда имеют progress, почти не оценивают.
3. rater — часто ставят оценки, редко пишут отзывы.
4. reviewer — ставят оценки и часто пишут отзывы.
5. collector — создают списки и добавляют много произведений.
6. social — активно подписываются и комментируют.
7. power_user — очень активные пользователи, делают почти всё.
8. moderator — обычная активность + действия модерации.

Примерные доли:
- inactive: 20–30%;
- observer: 15–25%;
- rater: 20–30%;
- reviewer: 8–12%;
- collector: 8–12%;
- social: 5–8%;
- power_user: 1–3%;
- moderator/admin: около 1%.

Для каждого пользователя создай внутренний профиль:
- activity_type;
- rating_count;
- review_probability;
- comment_count или comment_probability;
- list_count;
- follow_count;
- progress_count;
- favorite_content_kinds: movie/show/book/mixed;
- rating_bias: пользовательская строгость или щедрость;
- created_at.

Эти параметры не хранить в БД, они живут только внутри генератора.

Используй внешние данные по произведениям: external_rating_average, external_rating_count

Логика выбора произведений:
- Чем больше external_rating_count, тем чаще произведение должно попадаться в ratings, reviews, progress и list_items.
- Используй вес популярности примерно как log(1 + external_rating_count), чтобы очень популярные объекты не забрали все события.
- Учитывай favorite_content_kinds пользователя: киноманы чаще оценивают фильмы, читатели — книги, сериальные пользователи — эпизоды/сериалы.
- Иногда выбирай менее популярные произведения, особенно для collector и power_user, чтобы данные не были слишком однообразными.

Логика генерации оценки:
- ratings.value находится в диапазоне 0..3.
- Значение оценки должно зависеть от внешней средней оценки, пользовательского bias и случайного шума.
- Пример:
  normalized = external_rating_average / max_external_rating
  value = round(normalized * 3 + user.rating_bias + noise)
  value = clamp(value, 0, 3)
- Если шкала внешней оценки 0..10 — дели на 10.
- Если шкала 0..5 — дели на 5.
- Определи шкалу по данным или существующему коду импорта.

Соблюдай ограничения:
- Один пользователь не может иметь две оценки одного rateable.
- Отзыв создаётся только при наличии оценки этого же rateable этим же пользователем.
- Один пользователь может оставить максимум один отзыв на один rateable.
- Комментарий должен ссылаться на существующий post через parent_post_id.
- Подписка не может быть на самого себя.
- Дублирующие follows запрещены.
- progress должен ссылаться только на contents: movie, episode, book. Помечу, что в текущей реализации при выставлении оценки, также выставляется progress, так что учитывай это
- moderation_actions должны иметь ровно одну цель: либо target_post_id, либо target_list_id.
- Для списков сначала создавай rateable, потом list.
- Для оценок/отзывов работай через rateable_id произведения или списка.

Порядок генерации:
1. Убедиться, что roles содержат 3 роли: user, moderator, admin.
2. Создать или обновить 1000, а лучше чуть больше пользователей.
3. Назначить user_roles:
   - большинство user;
   - несколько moderator;
   - 1–2 admin.
4. Загрузить из БД все works/rateables/contents и внешние avg/count.
5. Сгенерировать ratings.
6. На основе части ratings сгенерировать posts-отзывы.
7. На основе существующих отзывов сгенерировать posts-комментарии.
8. Сгенерировать lists и соответствующие rateables.
9. Сгенерировать list_items.
10. Сгенерировать follows.
11. Сгенерировать progress.
12. Сгенерировать moderation_actions и при необходимости обновить is_hidden у posts/lists.

Нужна реалистичная временная ось:
- users.created_at распределить за последние 2–3 месяца;
- действия пользователя должны быть позже его регистрации;
- rating.created_at <= review.created_at;
- comment.created_at должен быть позже parent post;
- list_items.added_at должен быть позже lists.created_at;
- moderation_actions.created_at должен быть позже создания target post/list;
- updated_at должен быть не раньше created_at.

Сгенерируй тексты отзывов и комментариев шаблонно, но разнообразно.
Тональность отзыва должна зависеть от оценки:
- 0 — негативный;
- 1 — смешанный;
- 2 — положительный;
- 3 — восторженный.

Примеры смыслов отзывов:
- понравилась атмосфера;
- сильные персонажи;
- слабый темп;

- недооценено;

Распределение размеров списков:
- большинство списков: 3–15 элементов;
- часть списков: 20–50 элементов;
- редкие списки у collector/power_user: 100+ элементов.

Распределение комментариев:
- большинство отзывов без комментариев;
- часть отзывов получает 1–3 комментария;
- небольшая часть получает 4–15 комментариев;

Распределение подписок:

Цели подписок выбирай неравномерно: пользователи с большим числом отзывов, списков и подписчиков должны чаще получать новых подписчиков. Так появятся “локальные лидеры мнений”.

Progress:
- Генерируй только для contents.
- Для completed: value_now = value_max, если value_max известен.
- Для started: value_now < value_max.
- Для фильмов и эпизодов value_max можно брать из runtime_minutes, если доступно.
- Для книг используй число страниц, если есть; иначе реалистичный fallback, например 120–900.
- Если пользователь поставил оценку content-объекту, создай completed progress.
- Не создавай progress для show и season, если они не входят в contents.

Moderation:
- Создай минимум 1000 moderation_actions.
- Модераторами должны быть пользователи с ролью moderator/admin.
- Часть действий — hide, часть — restore.
- Цели: posts и lists.
- Если action = hide, обновляй is_hidden = true у цели.
- Если action = restore, обновляй is_hidden = false, но только для уже скрытых целей или реализуй последовательность hide -> restore.
